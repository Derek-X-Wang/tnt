# Voice Actions: a narrow Executor, not Computer Use

TNT will let the user turn voice into OS action — "send it" (paste the Rewrite into the target Worker Agent + Return), "open Cursor", "focus the tnt window". The reference point was **Codex Computer Use**, where the model autonomously sees and drives the GUI in a perceive→act→verify loop. **TNT deliberately does not build that.** Instead: a **closed allowlist** of scoped actions, performed by a concrete `MacActionExecutor`, invoked as tools live inside a Voice Turn. We call the feature **Voice Action**, not Computer Use.

This keeps the locked identity (CONTEXT.md: the Personal Master Agent **mediates** voice ↔ agents; "not a chatbot", does not execute beyond a clipboard handoff) while still delivering the Codex *feel* — you say it, it happens, it speaks back in one voice. The "true voice experience" and "TNT is not an autonomous actor" turned out to be **orthogonal**: the magic is low-latency in-conversation action (which we already proved with Appshot/ADR-0006: Realtime tool-call → TNT does the work → re-inject → one voice), not the autonomy.

## Decisions

1. **Narrow allowlist, not autonomous GUI driving.** v0 ships a `MacActionExecutor` over a **closed enum** of actions — `activateApp`, `focusWindow`, `pasteText`, `pressReturn` first; `type(String)` and `keystroke` later (see decision 5). Never a free-form `run(String)`. The autonomous perceive→act→verify loop (the only part that genuinely needs Codex-style Computer Use — "keep retrying until the test is green") is **Beyond v0**, grown behind the decider seam (decision 3), not by making the executor smart.

2. **The performer is a `final class`, not a protocol; ADR-0003 is untouched.** The component that performs actions — `MacActionExecutor` (NSWorkspace + CGEvent + AX) — is **permanent client**, the same category as `HotkeyHost` and `AccessibilityClient`: synthesizing input events requires local OS access and can never migrate to `tnt-server` (a server can _decide_ an action; only the client can _perform_ it). So it ships concrete, not behind a `protocol`. This keeps ADR-0003's signal intact — "every `protocol` is a thing whose v1 home is the server" stays literally true — rather than overloading `protocol` to also mean "test seam." The seam we actually need lives on the **decider** side, which is already a protocol (`CognitiveEngine`, server-future). Testing follows the PRD's rule for Cocoa-bound surfaces: the pure action-resolution logic (enum selection, target binding, confirmation tiering) is unit-tested; the CGEvent/AX-touching `MacActionExecutor` is smoke-tested + manual dogfood. (An earlier draft proposed `protocol Executor` *refining* ADR-0003 to "protocol = any seam, server-future or not"; rejected on a second opinion — splitting decider from performer is cleaner than weakening the rule, and a class that clicks/types/focuses muddies the "protocols move to the server" signal a reader relies on.)

3. **The Cognitive Engine decides; the Executor performs.** The `ExecutorAction` enum is the contract between *decider* and *hands*. In v0 the Realtime model calls executor tools directly. The intended future is that the **Cognitive Engine** (server-future) becomes the decider — choosing which action, and eventually planning a sequence — while `MacActionExecutor` (permanent client) stays a dumb, safe performer of one action at a time. This is the migration path toward more capability: **autonomy grows behind the Cognitive Engine seam (server-side), never by making the executor smart.** A future closed-loop "fix it until it works" is built by letting the Cognitive Engine perceive (via Appshot) between executor actions — a deliberate lifting of the firewall in decision 4, on the server side, not an executor that puppeteers.

4. **No perception feedback to executor actions — the structural firewall.** Executor actions return only a terminal result (success/failure), **never a screenshot**, to the model. Only an **Appshot** returns vision. With no perception feedback, the model cannot close a control loop with the GUI, so it cannot drift the narrow allowlist into de-facto puppeteering by chaining `focus → paste → "now click the thing"`. This is the enforced boundary between Voice Action (v0) and autonomous Computer Use (Beyond v0). Lifting it is exactly how the closed loop gets built later — deliberately, through the Cognitive Engine (decision 3).

5. **Defer the high-risk verbs.** Ship `activateApp`/`focusWindow`/`pasteText`/`pressReturn` first. `pasteText` is atomic and reviewable (set pasteboard + ⌘V); `type(String)` synthesizes N keystrokes and is the worst footgun on a focus race (wrong window = typing into a DM or a shell). Add `type`/`keystroke` only after the target guard (decision 6) is proven, gated behind the confirmation tier (decision 7).

6. **Every action is bound to an intended target and verified before firing.** This is the sharpest failure mode — the **target-window race**, the same bug class as the Realtime append/commit ordering race (issue #27). Between the tool-call firing and `MacActionExecutor` acting, the frontmost app can change (a notification steals focus, the user ⌘-Tabs, a dialog pops). The mitigation mirrors Appshot's "freeze context at capture": capture the target (bundle ID + window) at tool-call time, and `perform` **verifies the frontmost still matches immediately before acting** — otherwise it aborts and returns `target_changed`, and TNT speaks "Cursor isn't frontmost anymore, want me to focus it first?" Target-binding is a first-class part of `perform`, not an afterthought. The structured result is one of `done` / `needs_confirmation` / `target_changed` / `permission_missing` / `unsupported`.

7. **Confirmation by blast radius, not per-action.** Confirmation fatigue would kill the experience. Reversible actions (`activateApp`, `focusWindow`) just happen and TNT says what it did. Irreversible/exfiltrating actions (`pasteText`, `pressReturn`, `type`) require one spoken confirm per turn (TNT reads back the Rewrite → user says "send it"), never per keystroke.

8. **Term: "Voice Action," not "Computer Use."** Codex's term sets an expectation of full GUI autonomy we are deliberately not meeting; a TNT-native term keeps the mediator identity honest and resists the synonym drift CONTEXT.md already guards against.

## Why

- The user framed "true voice experience like Codex" against "separate actor is good code structure" as a trade-off. They aren't one: "separate actor" is a code-structure decision (a dedicated `MacActionExecutor` with a closed-enum contract), "true voice experience" is a UX decision (call executor tools live in the Voice Turn). Doing both gives the Codex feel without the autonomy.
- Full GUI puppeteering (the rejected option) would (a) contradict the locked mediator identity, (b) duplicate what Worker Agents already do, (c) explode the permission/safety/perception-loop surface, and (d) put an autonomous click-loop on TNT's plate when its value-add is voice mediation.
- A second opinion (Codex, via `counsel`) independently reached the same architecture and supplied decisions 2, 4, and 6.

## Considered alternatives

- **TNT delegates, never drives** (pure mediator; voice → instruction dispatched to a Worker Agent). Cleanest identity fit, but leaves the daily-value plumbing ("send it", "open Cursor") undone — the user explicitly wants those.
- **TNT drives the GUI directly, Codex parity** (CGEvent/AX/ScreenCaptureKit perceive→act loop, model decides clicks). Most powerful, but the identity break + safety surface + Worker-Agent overlap make it wrong for v0. Preserved as a Beyond-v0 path that grows behind the **Cognitive Engine** decider seam (lifting decision 4's firewall server-side) — a decider swap, not a re-architecture of the performer.

- **A "semantic handoff" path** — for Worker Agents, prefer app-native/agent-native channels (CLI, MCP, deep link, a future Worker-Agent transport) over driving the GUI; reserve OS-event synthesis for genuinely GUI-only targets. Not an alternative to the Executor so much as a sibling: when a structured channel to the target exists, route through it instead of `pasteText`+`pressReturn`. Folded into the Beyond-v0 `WorkerAgentDispatcher` direction rather than v0 scope.

## Consequences

- **Pasteboard asymmetry.** ADR-0004 bans *reading* the pasteboard for context. `pasteText` *writes* it. The posture becomes: **TNT may write the pasteboard to perform a Voice Action, but never reads it for context.** Recorded as an ADR-0004 amendment.
- **Issue #30 (Realtime function-call wiring) is on the Voice Action critical path** — the same dependency as Appshot. Executor tools ride the function-call → re-inject path.
- **First Voice Action = "send the Rewrite."** It closes the M1 clipboard-handoff gap (today: copy + notify + manual ⌘V↵), lives inside the existing target-app concept, and proves the whole Executor seam on a real daily path before the enum widens.
- The Voice Action milestone depends on M1 (Capture Set, `CognitiveEngine`, target-app capture) + #30. See roadmap.
